# HCDE 530 — project context

Plain-language context for this coursework repository. Use this with humans or AI assistants (e.g., Cursor) so help matches how you actually work.

## Who I am

- **Role:** UX design practitioner (human-centered design), **not** a software engineer.
- I’m comfortable with design craft and research-adjacent thinking; I want **practical** technical help, not production engineering by default.

## What this repository is for

- **Primary purpose:** **HCDE 530 class assignments** — organized so I can turn work in on time and reuse it as **portfolio** material.

## Who the work is for

- **Instructor / TAs** — must meet the rubric and read clearly.
- **Portfolio / public readers** — work should still make sense and look intentional when someone browses GitHub later.

Both matter; optimize for **clarity and credibility**, not internal-only shortcuts.

## What I expect to keep here

- **Mixed deliverables** over the quarter, including things like:
  - Python scripts (analysis, small utilities)
  - HTML/CSS (e.g., simple dashboards or static pages)
  - Data files (e.g., CSV)
  - Design artifacts (e.g., Figma exports) and written explanations where assignments ask for them

Assume **variety** week to week unless a specific assignment narrows it.

## Hardest constraint

- **Tight weekly deadlines.** Prefer solutions that are **fast to implement, easy to run, and easy to submit**, with minimal surprise. “Good enough for the rubric + portfolio” beats over-engineering.

## How I want technical help (default)

- **Checklists over commentary:** When I get code or technical changes, I want a **step-by-step**:
  1. **How to run it** (what to open, what command if any, what folder to be in).
  2. **How to submit it** (Git + Canvas flow below).
- I do **not** need heavy inline comments in code **unless I ask** — but a **short plain-language summary** of what changed is welcome.

## How I submit assignments

1. Push the relevant work to **GitHub** (public).
2. **Paste the GitHub link** (repo or specific file/path the instructions ask for) into **Canvas**.

So anything that touches “done” should be compatible with **“grader opens GitHub → understands what this is.”**

## Visibility and data

- Course repos and artifacts are intended to be **public** for grading and portfolio.
- Even with public repos, **avoid committing real participant PII or confidential work data** unless the course explicitly provides safe/synthetic data. When in doubt, use **synthetic or de-identified** examples for anything that looks like user research.

## Repository layout (current setup)

- **Course root:** `HCDE 530 Repository/` — may hold course-wide files.
- **`Week 1/`** is set up as its **own Git repository** for that week’s GitHub submission; the parent repo **ignores** `Week 1/` so the same files aren’t tracked twice. Future weeks may follow a similar pattern if useful.
